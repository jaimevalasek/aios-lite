Do Prompt ao Código
Sem Surpresas

Como o AIOSON transforma uma ideia vaga num sistema bem construído — usando agentes especializados, artefatos e gates de aprovação que impedem o AI de adivinhar o que você quis dizer.

Nível iniciante Leitura: ~12 min PT-BR
Contexto

O problema que o AIOSON resolve

Você já pediu para uma IA "criar um sistema de gerenciamento de estoque" e ela simplesmente começou a escrever código — sem perguntar nada, sem confirmar se era o que você queria, sem lidar com os casos especiais que você nem tinha pensado ainda?

Isso acontece porque a maioria das ferramentas de AI não tem protocolo. Elas recebem uma mensagem, geram uma resposta. Sem etapas, sem gates, sem memória entre sessões.

Sem protocolo
  • ⚡ Começa a codificar imediatamente
  • 🤔 Adivinha os requisitos não ditos
  • 🔄 Perde contexto após algumas mensagens
  • 💥 Reescreve tudo quando você muda a ideia
  • 🔀 Mistura decisão de produto com código
Com AIOSON
  • 📋 Documenta os requisitos antes de codar
  • ✅ Gates explícitos impedem avanços prematuros
  • 💾 Artefatos salvam estado entre sessões
  • 🎯 Cada agente tem escopo definido
  • 🔍 Ambiguidades viram decisões documentadas
Instalação

Começar em 3 passos

O AIOSON é um CLI Node.js. Você instala uma vez e configura em qualquer projeto novo ou existente.

1

Instale o AIOSON globalmente

Só precisa fazer isso uma vez na sua máquina. Requer Node.js 18+.

npm install -g @jaimevalasek/aioson

Ou use sem instalar, direto com npx:

npx @jaimevalasek/aioson init meu-projeto
2

Inicialize no seu projeto

Para um projeto novo, use init. Para um projeto já existente, use install.

# Projeto novo aioson init meu-projeto # Projeto existente cd meu-projeto aioson install .

Um wizard interativo vai perguntar qual AI usar (Claude Code, Codex, Gemini…), o idioma dos agentes e se quer ativar Squads.

3

Abra o AI e ative o primeiro agente

O AIOSON cria os arquivos de configuração que ensinam o AI como se comportar. Abra seu cliente AI e execute:

# Ver quais agentes estão disponíveis aioson agents # Pegar o prompt de ativação do setup aioson agent:prompt setup --tool=claude

Cole o prompt gerado na primeira mensagem do seu AI. O agente /setup vai mapear o contexto do projeto antes de qualquer trabalho começar.

💡

O que o install faz por baixo: cria uma pasta .aioson/ no seu projeto com agentes, skills, regras e configuração. Também cria um CLAUDE.md (ou equivalente) que instrui o AI sobre o protocolo AIOSON.

Spec-Driven Development

Como o SDD funciona na prática

Spec-Driven Development (SDD) é o protocolo central do AIOSON. A ideia é simples: antes de escrever qualquer código, a especificação precisa estar clara e aprovada. Nenhum agente avança sem que a fase anterior tenha entregado seu artefato.

🎯

A regra de ouro: um prompt vago vira uma especificação não-ambígua antes de qualquer código ser escrito. Nenhuma fase é pulada, nenhuma suposição fica silenciosa.

Cadeia de artefatos

Cada fase produz um documento concreto. O documento da fase anterior é a entrada obrigatória da próxima.

Fase 1
@product
prd-{slug}.md
Product Requirements Document. Define objetivos, o que está fora do escopo, comportamentos esperados e itens abertos.
→ Gates Gate A
Fase 2
@sheldon
sheldon-enrichment-{slug}.md
Enriquecimento do PRD. Áreas cinzentas decididas, extração de RF/RNF, score de prontidão, critérios de aceite endurecidos.
Fase 3
@analyst
requirements-{slug}.md
Requisitos com IDs (REQ-001), critérios de aceite verificáveis (AC-001), entidades e regras de negócio mapeadas.
Fase 4
@architect
architecture.md + design-doc-{slug}.md
Stack técnica, estrutura de módulos, dependências entre componentes, decisões não-óbvias documentadas com rationale.
→ Gate B
Fase 5
@dev
implementation-plan-{slug}.md
Sequência de execução, critérios de done por checkpoint, lista de arquivos a ler antes de cada fase.
→ Gate C
Fase 6
@dev / @deyvin
código + spec-{slug}.md (atualizado)
Implementação seguindo o plano aprovado. Spec atualizado com estado atual, último checkpoint e must_haves.
Fase 7
@qa
Verificação 4-tier
Existe → Substancial (não é stub) → Conectado (imports, registros) → Funcional (testes passando).

O @sheldon não apenas enriquece o PRD — ele avalia o tamanho do escopo e decide automaticamente a estrutura de entrega. Quanto maior e mais complexo o PRD, mais granular fica a decomposição.

Status e rastreamento

Como o AIOSON rastreia o progresso

Cada artefato tem um ciclo de vida explícito. O spec-{slug}.md é a memória viva da feature — ele registra em qual fase o trabalho está, o que foi decidido, e exatamente onde parou. É o que permite retomar uma sessão sem re-ler tudo.

Ciclo de vida do plano de implementação

○ draft
→
✓ approved
→
◆ in_progress
→
✓ completed

O implementation-plan.md começa como draft, só vira approved com confirmação explícita do usuário, e só pode ser executado depois disso. Sem aprovação, o @dev não começa.

Phase gates — o estado de cada fase

O arquivo spec-{slug}.md mantém um bloco YAML com o estado atual de cada gate:

---
feature: "stock-management"
classification: "MEDIUM"

phase_gates:
  requirements: approved  # Gate A passou — requisitos bloqueados
  design: approved      # Gate B passou — arquitetura bloqueada
  plan: pending        # Gate C ainda aguarda aprovação

last_checkpoint: "api: GET /products done — next: POST /products"

pending_review:
  - "Confirmar: password reset expira em 1h ou 24h?"

gray_areas_decided:
  - "2026-04-01 Soft delete em produtos — mantém histórico de pedidos"
---

Campos do spec — o que cada um significa

O @deyvin lê estes campos na ordem exata ao retomar uma sessão:

phase_gates.{fase}: pending
Fase ainda não aprovada. @dev não pode avançar para fases que dependem dela. Em MEDIUM é bloqueante; em SMALL é informativo.
phase_gates.{fase}: approved
Fase aprovada e bloqueada. Decisões desta fase são finais — @dev não reabre a discussão.
last_checkpoint
A primeira coisa que @deyvin lê ao retomar. Formato: "{fase}: {o que está feito} — next: {próximo passo}". Uma linha, acionável.
pending_review
Itens que precisam de verificação humana antes da próxima fase. @deyvin os lista ao usuário antes de continuar.
gray_areas_decided
Cada ambiguidade que foi levantada e decidida, com data e rationale. Evita que a mesma discussão aconteça duas vezes.
must_haves
Tripé de verificação: truths (comportamentos verificados), artifacts (arquivos não-stub), key_links (imports/registros conectados).

Gates de aprovação

Os 4 gates que protegem o projeto

Gates são pontos de verificação obrigatórios entre as fases. Em projetos MEDIUM, são bloqueantes — o agente não pode avançar sem aprovação explícita. Em projetos pequenos, são informativos.

A

Gate A — Requisitos aprovados

Entre @product/@analyst → @architect ou @dev

  • Objetivos claros e não-ambíguos
  • Comportamentos esperados descritos (não só features)
  • Restrições explícitas — o que esta versão NÃO faz
  • IDs de requisito: REQ-{slug}-{N}
  • Critérios de aceite independentemente verificáveis: AC-{slug}-{N}
B

Gate B — Design aprovado

Entre @architect → @dev

  • Abordagem técnica escolhida e documentada
  • Estrutura de módulos/pastas definida
  • Dependências entre componentes explícitas
  • Rationale para escolhas não-óbvias
  • @dev pode começar sem inventar lógica de negócio
C

Gate C — Plano aprovado

Antes de @dev/@deyvin executar um batch significativo

  • Sequência de execução definida
  • Checkpoints com critérios de done
  • Pacote de contexto listado (quais arquivos ler)
  • Decisões pré-tomadas são FINAIS — @dev não re-discute
D

Gate D — Verificação de execução

Antes de marcar uma fase ou feature como completa

  • Existe: o arquivo/função/componente existe de fato
  • Substancial: não é um stub vazio
  • Conectado: imports, registros e middleware estão ligados
  • Funcional: testes passando ou verificação manual documentada
⚠️

Um gate nunca é assumido como passado silenciosamente. Cada agente, ao final da sessão, declara explicitamente: "Gate A passou — ative @architect" ou "Gate A bloqueado — [razão]. Resolva antes de continuar."

Memória persistente

As 3 camadas de memória do projeto

À medida que o projeto evolui, o AIOSON acumula conhecimento em três camadas distintas dentro de .aioson/. Cada camada tem um propósito e cadência de atualização diferente — carregar a camada errada para a situação errada é uma das formas mais comuns de contexto desperdiçado.

1

rules/ — Convenções comportamentais

.aioson/rules/*.md
Estável · muda raramente

São overrides obrigatórios para os agentes. Quando uma regra existe em rules/, ela substitui o comportamento padrão do agente em todas as sessões — sem precisar repetir a instrução.

Uma convenção precisa ser aplicada em toda sessão de implementação
Um padrão de @dev apareceu 3+ vezes e deve virar permanente
O projeto usa um padrão diferente do default dos agentes
---
name: monetary-values-convention
description: Nunca usar float para valores monetários — usar inteiros em centavos
agents: [dev, architect]
priority: 10
version: 1.0.0
---

Sempre armazenar valores monetários como INTEGER (centavos).
Nunca usar FLOAT, DECIMAL ou string para preços.
2

docs/ — Conhecimento de domínio

.aioson/docs/*.md
Atualiza com o sistema externo

Documentação técnica que múltiplos agentes precisam ao longo do projeto — comportamento de APIs externas, quirks de integrações, modelos de dados que só existem em produção.

Agentes carregam um doc quando o description no frontmatter é relevante para a tarefa atual. Não é injetado automaticamente — o agente decide.

2+ features precisam do mesmo contexto externo (ex: Stripe webhooks)
Uma integração tem comportamento não-óbvio que já causou bugs
Um modelo de dados existe só em produção e não deriva do código
---
description: "Stripe webhook events — idempotency, retry behavior, event order"
scope: global
agents: [dev, architect]
---

# Stripe Webhook Context
Events can arrive out of order. Always check event.created
before updating state. Idempotency key: event.id.
3

context/design-doc*.md — Decisões por feature

.aioson/context/design-doc-{slug}.md
Atualiza a cada feature

Documento vivo de decisões da feature atual. Escrito pelo @discovery-design-doc, atualizado pelo @dev ao fechar cada feature. Decisões são append-only — nunca apagadas.

Um design-doc por feature (design-doc-stock.md) ou um global (design-doc.md) para decisões de arquitetura do projeto inteiro.

Documentar o escopo e decisões de uma feature específica
Registrar o que foi decidido durante a sessão atual
Rastrear itens com decisão ainda pendente
# Design Doc — Stock Management

## Decisions already made
- 2026-04-01 Soft delete — mantém histórico de pedidos
- 2026-04-02 UUID em vez de auto-increment — multi-tenant

## Decisions still pending
- [ ] Cache de estoque: Redis ou in-memory?

Onde vai cada informação?

Guia rápido para não colocar coisa no lugar errado:

Situação Onde guardar
Enforçar uma convenção de código no projeto inteiro rules/
Agentes precisam sempre saber como uma API externa funciona docs/
Decisões de escopo e arquitetura de uma feature específica design-doc-{slug}.md
Decisão de arquitetura que afeta o projeto inteiro design-doc.md (global)
Um padrão do @dev que apareceu 3+ vezes rules/ (promover via @dev)
Estado atual de implementação de uma feature spec-{slug}.md
Requisitos e critérios de aceite requirements-{slug}.md
Contexto geral do projeto (stack, times, objetivos) context/project.context.md
🔄

Como as camadas interagem: Rules são carregadas sempre. Docs são carregados quando relevantes. Design-docs são carregados por feature. O resultado é que cada agente começa a sessão com exatamente o que precisa — nem mais, nem menos.

Aprendizagem adaptativa

Brains — memória procedural dos agentes

Regras descrevem o que fazer. Docs descrevem como algo funciona. Brains descrevem o que funciona de verdade — padrões com qualidade comprovada, anti-padrões conhecidos, e conexões entre conhecimentos.

Inspirado no Zettelkasten: cada nó de brain tem uma qualidade de 1–5 baseada em uso real, um verdict, e links para nós relacionados. Agentes carregam brains sob demanda — o índice é sempre carregado (~2KB), os arquivos de brain só quando as tags batem.

Verdicts — como o agente avalia cada padrão

EXCELLENT
Melhor da categoria
Use por padrão. Verificado em produção com resultado excelente. Score 5.
GOOD
Sólido, use quando EXCELLENT não se aplica
Funcionou bem em contextos reais. Score 4.
BEST_PRACTICE
Linha de base obrigatória
Sempre incluir — é o piso de qualidade, não o teto.
AVOID
Anti-padrão — nunca usar
Mantido no brain para documentar o que NÃO fazer. Score 1.
BROKEN
Quebrado no contexto atual
Funcionou antes mas está quebrado no stack/versão atual.

Anatomia de um nó de brain

{
  "id": "css-001",
  "title": "Hover animation com transform+opacity",
  "tags": ["css", "hover", "animation"],
  "q": 5,                       // qualidade 1-5
  "v": "EXCELLENT",              // verdict
  "s": "transform+opacity é GPU-accelerated — nunca usar top/left",
  "p": "transition: transform 0.2s, opacity 0.2s;",
  "not": "transition: all — anima propriedades que forçam reflow",
  "warn": "Testar no Safari — transform-origin pode diferir",
  "see": ["css-002", "css-007"]    // rede Zettelkasten
}

Ciclo de aprendizagem

Brains evoluem com cada projeto. Ao fechar um ciclo de desenvolvimento, os padrões que funcionaram são promovidos:

1

Sessão finalizada

@dev completa uma fase e registra as decisões tomadas no spec-{slug}.md.

2

Evolução de padrões

O comando aioson learning:evolve destila padrões das sessões completadas e adiciona nós ao brain com qualidade inicial.

aioson learning:evolve aioson learning:apply
3

Promoção para rules

Se um padrão apareceu 3+ vezes e funcionou bem (q ≥ 4), o agente sugere promovê-lo para rules/ — virando convenção permanente do projeto.

4

Consulta sob demanda

Em sessões futuras, agentes consultam o brain quando as tags da tarefa batem com os nós disponíveis:

# Padrões excelentes sobre CSS animation node .aioson/brains/scripts/query.js \ --tags css,animation --verdict EXCELLENT # Anti-padrões a evitar node .aioson/brains/scripts/query.js --avoid

Contexto inteligente

O AI nunca perde o fio da meada

Um dos maiores problemas com AI em projetos reais: o contexto acaba, a sessão crasheia e você precisa repetir tudo. O AIOSON tem um sistema completo para isso.

🔄

Session Recovery

Quando o Claude compacta ou a sessão cai, um comando restaura o estado completo em menos de 2.000 tokens.

aioson recovery:generate aioson recovery:show
📊

Context Monitor

Barras de uso em tempo real com limites adaptativos por tamanho de projeto.

aioson context:monitor # ████████████░ 73% ⚠ warning
🔍

Busca FTS5

Encontra qualquer informação em todos os artefatos do projeto em milissegundos.

aioson context:search "webhook retry"
✂️

Agent Sharding

Carrega só as instruções relevantes para o objetivo atual — 68% menos tokens por agente.

aioson agent:load deyvin --goal="fix modal"

Squads

💻

Squad de Software

🎬

Squad de Conteúdo

⚖️

Squad Jurídico

# Criar um squad aioson squad:create content-squad # Rodar o squad como daemon em background aioson squad:daemon content-squad # Dashboard em tempo real aioson squad:dashboard # → http://localhost:4180

Cada squad tem seu próprio Genome — uma camada cognitiva que define não só o que o agente faz, mas como ele pensa. Você pode até importar o perfil cognitivo de uma pessoa real (como Alex Hormozi ou um especialista do seu setor) e aplicar ao squad.
