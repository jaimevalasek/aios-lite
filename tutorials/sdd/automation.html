SDD Automation Scripts

12 scripts determinísticos que eliminam 4.800–8.800 tokens por feature, movendo verificações de estado, validação de artefatos e gate checks para fora do contexto LLM.

12 comandos novos
Zero chamadas LLM
Regra dos 80%
Motivação

Por que 80% do trabalho não precisa de LLM

Antes desses scripts, perguntas como "o gate A está aprovado?" ou "qual a classificação dessa feature?" eram resolvidas dentro da janela de contexto do agente — consumindo tokens desnecessários para trabalho determinístico puro.

Antes (sem scripts)

O agente lê project.context.md, spec-checkout.md, implementation-plan.md e mais 3 arquivos para descobrir o que já existe — usando 3.000+ tokens no processo.

Agora (com scripts)

Uma chamada a aioson preflight . retorna tudo em JSON estruturado em < 200ms. O agente usa o resultado diretamente.

💡

Regra dos 80%: 80% do trabalho de verificação, estado e validação é determinístico — scripts fazem melhor, mais rápido e sem custo de tokens. Reserve o contexto do agente para os 20% que exigem raciocínio.

Visão Geral

Os 12 scripts e o que cada um faz

Cada script elimina um tipo específico de trabalho que os agentes faziam manualmente.

aioson preflight
Pré-voo completo
Coleta modo, classificação, framework, test runner, artefatos, gates e prontidão em uma chamada.
-1.800–3.200 tokens
aioson classify
Classificação MICRO/SMALL/MEDIUM
Scoring automático via PRD + requirements. Modo interativo disponível.
-800–1.500 tokens
aioson sizing
Modelo de sizing
Decide entre inplace, phased_inplace e phased_external contando entidades e integrações.
-400–800 tokens
aioson detect:test-runner
Detecção de test runner
PHPUnit, Jest, Vitest, Pytest, RSpec, Forge, node:test — detecta por arquivos de config.
-300–600 tokens
aioson gate:check
Verificação de phase gate
Valida pré-requisitos e artefatos dos gates A/B/C/D. Retorna PASS ou BLOCKED.
-600–1.200 tokens
aioson artifact:validate
Cadeia de artefatos
Verifica PRD → spec → plano → conformance e indica o próximo item faltante.
-500–900 tokens
aioson pulse:update
Atualização de pulse
Cria ou atualiza project-pulse.md com agente ativo, feature, gate e próximo passo.
-150–300 tokens
aioson state:save
Ponto de continuação
Salva fase, status e spec-version em dev-state.md com histórico auditável.
-150–300 tokens
aioson feature:close
Fechamento de feature
Atualiza spec (QA sign-off), features.md e project-pulse.md com um comando.
-800–1.400 tokens
aioson workflow:execute
Orquestração de feature
Monta e executa o plano de agentes por tier. Suporta --dry-run e --start-from.
economiza coordenação manual
aioson runner:queue:from-plan
Enfileirar fases do plano
Extrai fases ## Phase N: e enfileira no runner com prioridades correspondentes.
automação de runner
aioson learning:auto-promote
Promoção de aprendizados
Promove aprendizados process/quality de alta frequência para .aioson/rules/.
memória procedural automática

Tutorial 1 de 6

Começar uma sessão com preflight

O comando preflight substitui a leitura manual de 4–6 arquivos de contexto. Use-o sempre que iniciar uma sessão de agente.

# Verificar prontidão antes de começar o @dev aioson preflight . --agent=dev --feature=checkout --json
⚡

O campo readiness.status retorna READY ou BLOCKED com lista de blockers. Se BLOCKED, resolva os blockers antes de iniciar o agente.

Output
JSON
mode, classification, framework
Como o projeto está configurado e qual tier de feature usar
Output
JSON
artifacts, phase_gates
Quais arquivos já existem e quais gates foram aprovados
Output
JSON
readiness: READY / BLOCKED
Veredito final com lista de blockers se houver
use antes de qualquer sessão
Tutorial 2 de 6

Verificar gates com gate:check

Use antes de avançar para o próximo agente. Cada gate tem pré-requisitos e artefatos obrigatórios.

A
Gate A — Requirements
requirements-{slug}.md ou prd-{slug}.md
sem pré-requisitos
B
Gate B — Design
spec-{slug}.md
requer Gate A
C
Gate C — Plan
implementation-plan-{slug}.md
requer Gates A + B
D
Gate D — Execution
QA sign-off em spec + gate_execution
requer Gates A + B + C
# Antes de acionar @dev: verificar Gate C aioson gate:check . --feature=checkout --gate=C --json # Aceita também nomes completos aioson gate:check . --feature=checkout --gate=plan

Se o resultado for PASS — avance para o próximo agente.
Se for BLOCKED — o campo blockers indica exatamente o que falta.

Tutorial 3 de 6

Salvar estado durante o dev

Use state:save e pulse:update ao fim de cada fase para garantir recuperação de sessão sem perda de contexto.

1

Salvar ponto de continuação

Registra fase, status e spec-version em dev-state.md com entrada histórica.

aioson state:save . \ --feature=checkout \ --phase=2 \ --status=in_progress \ --next="Phase 3: webhook" \ --spec-version=4
2

Atualizar o pulse

Registra em project-pulse.md o agente ativo, a feature e o próximo passo.

aioson pulse:update . \ --agent=dev \ --feature=checkout \ --action="Phase 2 concluída" \ --next="Phase 3: webhook handler"
3

Na próxima sessão: ler estado

O preflight já inclui dev_state e pulse no JSON. Não é necessário ler os arquivos manualmente.

aioson preflight . --agent=dev --feature=checkout --json # → dev_state.phase = 2, dev_state.next = "Phase 3: webhook"
Tutorial 4 de 6

Fechar feature após QA com feature:close

Um comando substitui a edição manual de três arquivos: spec-*.md, features.md e project-pulse.md.

# PASS — feature aprovada pelo QA aioson feature:close . \ --feature=checkout \ --verdict=PASS # PASS com residual (item aceito mas não testado completamente) aioson feature:close . \ --feature=checkout \ --verdict=PASS \ --residual="Email delivery não testado E2E" # FAIL — feature rejeitada aioson feature:close . \ --feature=checkout \ --verdict=FAIL \ --notes="Auth edge case ausente"
Atualiza
spec file
spec-checkout.md
Adiciona bloco ## QA Sign-off + atualiza gate_execution no frontmatter
Atualiza
features
features.md
Muda status para done ou qa_failed com data
Atualiza
pulse
project-pulse.md
Limpa active_feature, registra last_gate: Gate D
Tutorial 5 de 6

Orquestrar sem dashboard com workflow:execute

Executa o fluxo completo de agentes de uma feature, passando o contexto correto para cada um e registrando estado ao longo do caminho.

1

Dry-run: ver o plano antes de executar

aioson workflow:execute . \ --feature=checkout \ --classification=SMALL \ --dry-run \ --json # → steps: [{agent:"product"}, {agent:"analyst"}, {agent:"dev"}, {agent:"qa"}] # Steps com skip:true já têm artefatos — serão pulados automaticamente
2

Executar o fluxo

aioson workflow:execute . \ --feature=checkout \ --tool=claude
3

Retomar de um ponto específico

Se a feature já passou por product e analyst, use --start-from para pular:

aioson workflow:execute . \ --feature=checkout \ --tool=claude \ --start-from=dev
Tutorial 6 de 6

Fluxo completo de uma feature SMALL

Do início ao fechamento, usando apenas scripts SDD — sem leitura manual de arquivos de contexto.

# ── 1. Pré-voo ─────────────────────────────────────────────── aioson preflight . --agent=product --feature=checkout # ── 2. Classificar ─────────────────────────────────────────── aioson classify . --feature=checkout # → SMALL # ── 3. Verificar artefatos antes de iniciar @analyst ───────── aioson artifact:validate . --feature=checkout aioson gate:check . --feature=checkout --gate=A # ── 4. Executar workflow (product → analyst → dev → qa) ─────── aioson workflow:execute . --feature=checkout --tool=claude --dry-run aioson workflow:execute . --feature=checkout --tool=claude # ── 5. Durante @dev: salvar ao fim de cada fase ─────────────── aioson state:save . --feature=checkout --phase=2 \ --status=in_progress --next="Phase 3: webhook" aioson pulse:update . --agent=dev --feature=checkout \ --action="Phase 2 done" --next="Phase 3" # ── 6. Antes de @qa: checar Gate D ─────────────────────────── aioson gate:check . --feature=checkout --gate=D # ── 7. QA sign-off ─────────────────────────────────────────── aioson feature:close . --feature=checkout --verdict=PASS # ── 8. Promover aprendizados para regras ───────────────────── aioson learning:auto-promote . --threshold=3
🚀

Todos os oito passos substituem a leitura e edição manual de 10+ arquivos de contexto. Nenhuma chamada LLM é feita pelos scripts — apenas I/O de arquivo e SQLite.
