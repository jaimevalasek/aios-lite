## Plano 61 — Event Enrichment (Dashboard Visibility)

**Objetivo:** enriquecer o SQLite com dados que o dashboard precisa para mostrar estado real das runs.

### Mudanças em `src/runtime-store.js`

**5 novas colunas em `execution_events`** (via `ensureLegacyColumns` — backward compatible):
- `plan_step_id TEXT` — vincula evento a um passo específico do plano de implementação
- `worker_status TEXT` — status atual do worker no momento do evento (`idle`, `working`, `blocked`)
- `verdict TEXT` — veredito de QA (`PASS`, `FAIL`, `PARTIAL`)
- `token_count INTEGER` — tokens consumidos na chamada associada ao evento
- `progress_pct REAL` — percentual de progresso do step atual (0–100)

**4 novos índices:**
- `idx_execution_events_agent_type` — filtragem rápida por agente + tipo de evento
- `idx_execution_events_verdict` — busca de VERDICTs sem full scan
- `idx_plan_phases_status` — lookup de fases por status
- `idx_artifacts_run` — listagem de artefatos por run

**`appendRunEvent()` e `insertExecutionEvent()` atualizados** para aceitar e persistir os novos campos.

### Mudanças em `src/commands/live.js`

`runRuntimeEmit()` agora aceita 4 flags novas:
```bash
aioson runtime:emit . --agent=dev --type=plan_checkpoint --plan-step=FASE-1 \
  --worker-status=working --verdict=PASS --token-count=4200 --progress-pct=60
```

### Mudanças em `src/commands/runtime.js`

`runAgentDone()` agora aceita:
```bash
aioson agent:done . --agent=dev --summary="Cart implementado" \
  --verdict=PASS --artifacts="src/cart.ts,tests/cart.test.ts" --plan-step=FASE-2
```
- `--verdict` — registra veredito no evento de conclusão
- `--artifacts` — lista CSV de paths; cada um gera uma linha na tabela `artifacts`
- `--plan-step` — vincula o `agent_done` ao step do plano

## Plano 63 — Spec-Driven Sync

**Objetivo:** conectar os arquivos `spec-{slug}.md` ao SQLite automaticamente, eliminando o gap entre o que os agentes escrevem e o que o dashboard vê.

### Novos arquivos

#### `src/commands/spec-sync.js` → `aioson spec:sync`

Lê todos os `spec*.md` de `.aioson/context/`, sincroniza com SQLite:
- **Learnings** (seção `## Session Learnings`) → tabela `project_learnings` (upsert idempotente — `frequency++` se já existe)
- **Phase gates** (frontmatter `phase_gates`) → tabela `plan_phases`
- **last_checkpoint** → exibido no relatório

```bash
aioson spec:sync .
# Spec Sync — my-project
# spec-checkout.md
#   Learnings synced: 3
#   Plan phases updated: 1
# Summary: 3 learnings synced, 1 plan phases updated
```

#### `src/commands/spec-status.js` → `aioson spec:status`

Dashboard textual de features: fase atual, último agente, último checkpoint.

```bash
aioson spec:status .
# Feature             Phase     Status          Last Agent      Checkpoint
# checkout            2/5       in_progress     dev             Criando migration...
# Active learnings: 8  |  Promotable (freq≥3): 2
```

#### `src/commands/learning-export.js` → `aioson learning:export`

Exporta `project_learnings` como nodes Zettelkasten em `.aioson/brains/`.

```bash
aioson learning:export . --min-frequency=2
# 5 nodes written to .aioson/brains/
# 2 learning(s) with frequency ≥ 5 — run: aioson learning:evolve
```

#### `src/commands/spec-checkpoint.js` → `aioson spec:checkpoint`

Ponto de recuperação manual: lê `last_checkpoint` do spec e registra no SQLite.

```bash
aioson spec:checkpoint . --feature=checkout
# last_checkpoint: "Criando migration cart_items — step 3 of 5"
# Checkpoint registered: run_key: dev-1711234567890
```

## Plano 62 — Token Economy

**Objetivo:** dar visibilidade e controle sobre o consumo de tokens antes e durante sessões.

### Novos arquivos

#### `src/commands/context-health.js` → `aioson context:health`

Analisa `.aioson/context/`, estima tokens (1 token ≈ 4 chars), identifica arquivos pesados e specs stale.

```bash
aioson context:health .
# Context Health Report — my-project
# discovery.md          28KB      ~7,000  ⚠ HEAVY
# Total context load:   70KB      ~17,500 tokens
# ⚠ 2 stale spec file(s) — run: aioson context:trim .
```

#### `src/commands/context-trim.js` → `aioson context:trim`

Detecta specs de features `done`, seções com muitos bullets. Arquiva com `--force` (nunca deleta).

```bash
aioson context:trim . --dry-run
# Stale specs: spec-auth.md (auth is done, 8.2KB)
# [dry-run] Would archive 1 stale spec(s)

aioson context:trim . --force
# Archived: spec-auth.md → context/archive/spec-auth.md
```

### Atualização em `src/commands/context-monitor.js`

Adicionado modo de budget de projeto com 3 zonas:
- **safe** < 60% — verde
- **warning** 60–80% — amarelo, emite `context_budget_warning` no SQLite
- **critical** ≥ 80% — vermelho, emite `context_budget_critical` no SQLite

```bash
aioson context:monitor . --budget=80000 --tokens=52000
# ⚠ Context: 52,000 tokens (65%) — WARNING
# Suggestion: /clear before next agent activation
```

## Plano 64 — Devlog Auto-Pipeline

**Objetivo:** fazer devlogs escritos pelos agentes chegarem ao SQLite (learnings, artifacts, decisões, verdict) de forma automática.

### Template padronizado de devlog

Atualizado em `template/.aioson/config.md` e inline em `dev.md` / `qa.md`:

```markdown
---
agent: dev
feature: checkout
started_at: 2026-04-01T14:23:00Z
finished_at: 2026-04-01T15:45:00Z
status: completed
verdict: null
plan_step: FASE-1
---
## Summary
Implementei a migration de cart_items.
## Artifacts
- src/database/migrations/003_cart_items.ts
## Decisions
- UUID para cart_id — compatibilidade multi-sessão
## Learnings
- [process] Rodar `npm test -- --filter=cart` isola testes do módulo
- [domain] cart_items deve ter UNIQUE(cart_id, product_id)
## Blockers
- Nenhum
```

### Novos arquivos

#### `src/commands/devlog-process.js` → `aioson devlog:process`

Processa devlogs de `aioson-logs/devlog-*.md`:
- **Artifacts** → tabela `artifacts`
- **Decisions** → `execution_events` (tipo `decision`)
- **Learnings** (`[process]`/`[domain]`/`[quality]`) → `project_learnings` (upsert)
- **Verdict** → `execution_events.verdict`
- Marca devlog com `processed_at` no frontmatter (idempotente)

```bash
aioson devlog:process .
# devlog-dev-1711234567.md
#   Artifacts: 3 registered ✓
#   Learnings: 2 upserted ✓
# Processed: 2/3 devlogs  |  Malformed (skipped): 1
```

#### `src/commands/devlog-watch.js` → `aioson devlog:watch`

Daemon que observa `aioson-logs/` e processa automaticamente novos devlogs.
- Detecta WSL2 via `/proc/version` e usa polling de 5s automaticamente
- Fallback para polling se `fs.watch` falhar

```bash
aioson devlog:watch . &
# [DEVLOG WATCHER] WSL2 detected — using polling (5s interval)
# [14:47:08] New: devlog-qa-1711237890.md → processing...
# [14:47:09] Processed: devlog-qa-1711237890.md → 1 learnings, 1 artifacts → SQLite ✓
```

#### `src/commands/devlog-export-brains.js` → `aioson devlog:export-brains`

Wrapper sobre `learning:export` otimizado para o pipeline de devlogs (min-frequency=2 por padrão).

```bash
aioson devlog:export-brains . --min-frequency=3
# 3 nodes written to .aioson/brains/
# Run: aioson learning:evolve to promote high-frequency nodes to genome.
```
